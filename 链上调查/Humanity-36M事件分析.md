## 0 · 一句话结论
Humanity Protocol 的 $36M 被盗,**根因不在智能合约 —— 合约一行没坏**。问题是它的"多签"其实是**假多签**:控制代币与桥的 7 把私钥,备份全在**一台开发者笔记本**上。攻击者用一封伪装成 Bithumb 的钓鱼邮件拿下这台机器,就拿到了对以太坊和 BSC 基础设施的单方面控制权,跨两链共动用约 **447M $H**(约 $36M)。**这是审计救不了的一类攻击 —— 密钥管理与运营安全(opsec),才是真正的肋。**

## 1 · 背景
Humanity Protocol 是做"人格证明(Proof of Humanity)"的身份网络,$H 为其代币,部署在以太坊与 BSC,并通过**可升级代理(Proxy)+ ProxyAdmin** 管理合约、通过桥跨链。关键认知:**谁控制 ProxyAdmin / 升级权限,谁就能把合约换成任意逻辑、增发或转走代币。** 这套权限本应由真多签(多人、多机、物理隔离)守护。

## 2 · 根因:假多签 = 单点
- 项目的"多签"密钥(共 7 把,覆盖以太坊与 BSC 的 token/bridge ProxyAdmin)**备份在同一台开发者(某 director)的 Windows 机器上**。
- 一台机器被攻陷 = 全部权限失守。所谓"多签"在物理上退化成**单点**。
- **这不是合约漏洞,是密钥管理 + opsec 失败。** 合约审计无法发现"你的多签密钥其实都在一台笔记本上"。

> 教训核心:**多签的安全性 = 密钥的物理隔离程度**。把多把密钥放一台机器,等于花多签的钱买单签的安全。

## 3 · 攻击流程
1. **钓鱼投递**:一封伪装成韩国交易所 **Bithumb** 的邮件,带"代币锁仓表更新"恶意附件,投给 Humanity 的一名 director。
2. **拿下机器**:附件释放恶意软件,远程控制其 Windows 机,**提取 MetaMask 钱包数据与私钥**。
3. **以太坊端**:先从 admin 热钱包盗走约 **6M $H**;随后夺取**桥的 ProxyAdmin** 控制权、**升级 $H 合约**,从协议转出约 **141.18M $H**。
4. **BSC 端**:夺取 BSC 代币 ProxyAdmin(3 把 Safe owner 密钥),**增发约 300M $H**。
5. **合计**:跨两链约 **447M $H** 受影响;其中超过 **$21M** 已被换成 ETH 流向攻击者控制的钱包,部分换成 USDC 存入 KuCoin。

## 5 · 归因
Quantstamp 取证将本次攻击归于**朝鲜(DPRK)关联威胁组织**,依据包括:钓鱼诱饵(冒充 Bithumb 的"锁仓表"附件)、恶意软件行为特征,以及样本中出现的**韩国数字证书**等技术指纹 —— 与 DPRK 历次入侵手法一致。**先攻陷个人设备、再横向拿密钥**,是其典型剧本(同期 Bybit $1.5B 亦属此类)。

## 6 · 修复与防御建议
1. **真多签,物理隔离**:多签密钥必须分散在不同人、不同设备、不同地点;严禁多把密钥备份在同一台机器。硬件钱包 / HSM / 阈值签名(MPC)优先。
2. **升级权限上 timelock + 多签**:ProxyAdmin / 合约升级是最高危单点,必须配时间锁 + 真多签,给社区留出反应窗口。
3. **opsec 防钓鱼**:面向高权限人员的反钓鱼演练;高危操作机器与日常办公机隔离;邮件附件零信任。
4. **可升级 = 高危**:代理可升级带来灵活性,也带来"一把钥匙换掉全部逻辑"的单点风险,权限治理要与之匹配。
